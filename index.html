Hosted service

Identity and signaling for
devices behind NAT.

idr.to implements ID-based routing (IDR) — DNS-like names, cryptographic device identity, and policy-controlled access. The server handles registry and WebRTC signaling; your traffic stays peer-to-peer whenever the network allows.

Core capabilities

  • 01

    Stable addressing

    Reach any host with a canonical URI like idrto:example.com/laptop/22 or a synthetic *.idr hostname.

  • 02

    Crypto identity

    Each device gets a Device-ID derived from its Ed25519 key — no centrally assigned addresses.

  • 03

    Policy before packets

    Allowlist-based ACLs are evaluated at resolve time. Same-entity access works by default; cross-entity access requires explicit allows.

Features

Everything you need to connect fleets, APIs, and edge devices

  • Identity registry

    Entities, device identities, and host-to-device mappings. Delegate subdomains with signed X.509 certificates — like DNS, but for connectivity.

  • WebRTC signaling

    Persistent WebSocket from target agents; ephemeral sessions for sources. SDP offers, answers, and ICE candidates relayed through wss://idr.to/v1/signal.

  • Transparent L4 tunnels

    Multiplexed idr-tunnel-v1 over WebRTC data channels. SSH, Postgres, HTTP, and HTTPS work as raw TCP — no protocol termination.

  • STUN & optional TURN

    STUN assists NAT traversal out of the box. Add TURN relay when strict firewalls or symmetric NAT block direct peer paths.

  • Agents & CLI

    Run idr-agent in source, target, or hybrid mode. Manage entities, hosts, ACLs, and connections with the idr CLI.

  • Read-only portal

    Inspect certificates, delegations, host mappings, and ACLs in the web portal. Registry changes are made with the CLI.

Architecture

Control plane hosted. Data plane peer-to-peer.

idr.to is the registry, policy engine, and signaling relay. Bytes flow over WebRTC between agents whenever possible.

Source agent

Resolves target, passes ACL checks, exchanges SDP/ICE via signaling.

Target agent

Maintains persistent WebSocket; dials local services on incoming tunnels.

When P2P fails, optional TURN relay carries media through turn.idr.to. For legacy HTTPS clients that cannot use WebRTC, operators may deploy an optional HTTPS edge relay (*.idr.to DEF hostnames) — application bytes still terminate on the target agent.

Deployment

Three ways to reach a target

Most traffic uses WebRTC peer-to-peer tunnels. Add relay capacity only when the network or client stack requires it.

  • P2P (default)

    Agents exchange SDP/ICE via wss://idr.to/v1/signal. Data flows over idr-tunnel-v1 on a WebRTC data channel — SSH, Postgres, HTTP, and raw TCP.

  • TURN (optional)

    When strict NAT or firewalls block direct paths, media relays through turn.idr.to. Billed per entity as an add-on quota.

  • HTTPS edge relay (optional)

    For browsers and tools that speak ordinary HTTPS/TLS to a public hostname, operators can run the open-source https-relay stack with DEF-encoded *.idr.to names. The relay forwards encrypted bytes to the target agent; TLS terminates on the device.

Get started

Install the agent and register your first device

  1. Install

    curl -fsSL https://download.idr.to/install.sh | bash

    Also available via Homebrew, winget, apt, rpm, Chocolatey, and Scoop.

  2. Authenticate & register

    idr auth login
idr device register
idr agent run

  3. Connect

    idr connect idrto:example.com/laptop/22

    Or use synthetic *.idr hostnames with SSH, Postgres, and curl.

API discovery, signaling, and ICE endpoints are served from idr.to. Embed the same connectivity in mobile apps with the Flutter and C SDKs.
